# 📖 日次読書サマリー生成 — Routine 指示書

このファイルは、毎朝 07:00 JST に起動するクラウド Claude エージェント（Routine）の実行手順書。
エージェントはこのリポジトリ（oshomadesse/books-summary）のチェックアウト内で作業する。

## 前提・共通ルール

- 「今日」の日付は必ず `TZ=Asia/Tokyo date +%Y-%m-%d` で取得する（以下 `YYYY-MM-DD`）。
- **多重実行防止**: `100_Inbox/Books-YYYY-MM-DD.md` が既に存在する場合、本日分は生成済み。何もコミットせず終了する。
- LINE 通知はこのエージェントの仕事ではない。push を受けた GitHub Actions（`.github/workflows/line-notify.yml`）が `data/latest.json` を読んで送信する。
- 捏造禁止: Web 上で確認できた情報のみ使う。曖昧・不明な点は「不明」と書く。

## Step 1: 選書

1. `data/books_read.json`（既読リスト）を読む。
2. 以下の条件で「今日読むべき本」の候補を **5冊** 選ぶ:
   - カテゴリは **ビジネス / 自己啓発 / ライフスタイル / 心理学** のいずれか1つ
   - 日本語で出版されている**実在の書籍**（日本語タイトル）。翻訳書は日本語版タイトルで扱う
   - 小説・文学・ライトノベル・児童文学・短編・青春ものは除外
   - 既読リストのタイトルと同一・類似（表記ゆれ・翻訳違い含む）の本は除外
   - 直近の既読カテゴリに偏らないようバランスを取る
3. 候補5冊のうち実在が Web 検索で確認できたものから **1冊をランダムに選ぶ**。

## Step 2: Deep Research

選んだ本について WebSearch / WebFetch で徹底的に調査し、以下を日本語でまとめる:

- **核心的メッセージ**（本の主張を凝縮した1段落）
- **エグゼクティブ・サマリー**（全体像がわかる要約、3〜5段落相当）
- **主要概念・キーワード**（5〜8個、それぞれに噛み砕いた解説）
- **各章の要約**（章立てが不明なら論点ごとの疑似章を立てる）
- **具体例・ケーススタディ**（3件以上、背景と示唆つき）
- **重要な引用**（原典の引用が取れない場合は要旨で代替し「要旨」と明記）
- **今日できるアクション**（15〜30分で実行できる具体行動を3つ、各に目的と期待効果）
- **関連書籍**（3〜5冊、「書名（著者）: 関連性」の形式）

## Step 3: インフォグラフィック生成

Step 2 の調査内容を、以下の指示（旧システムから継承した確定仕様）に従って
**単一の自己完結 HTML** に変換する:

> 【Infographic指示（最終版）】
> あなたは視覚化のプロのデザイナー兼フロントエンド実装者です。書籍リサーチの内容をすべて読み込み、書籍を「本を読んでいない人でも全体像が掴める」単一HTMLインフォグラフィックに変換してください。
>
> タブ構成（固定）：
> - 主要概念の詳細：初見でも理解できるよう噛み砕いた説明をカード形式で整理し、各概念に「理由（なぜ重要か）」を明示する。
> - 各章の要約：各章のポイントを短い見出し＋箇条でまとめ、章ごとに「理由」を付す。
> - 具体例：核心メッセージが伝わる実際のケースやメタファーを3件以上、背景・示唆・「理由（この例が示すこと）」つきで提示する。
> - 重要な引用：引用文＋一言補足で並べる（引用がない場合は要旨で代替し"要旨"と明記）。
> - 今日のアクション：15〜30分で実行できる行動を3つ以上、目的と期待効果つきで提示する。
>
> 必須要件：
> 1. テキストはタブ／カード／列で分割。各カードは「見出し（1行）」＋「要点箇条（3〜5行）」。長文の段落は禁止。
> 2. ファーストビューに「要約=問い×答え×根拠（Why & How）」の専用ブロックを配置する。
> 3. 重要な数字・指標は視覚化（横棒／円グラフ／数値バッジ等）で表現する。
> 4. カラーパレットは「プライマリー / セカンダリー / アクセント」の3色（グラデーション可）。カラーコードをHTML内コメントに明記する。
> 5. リサーチ内容は抜けなく反映し、図や絵文字などの視覚要素を多用する。
> 6. スマホ表示最優先のレスポンシブ設計。タブ切替は JavaScript、適切な `aria-` 属性を付与する。
> 7. 外部 CDN・外部画像は禁止（全てインラインで完結）。先頭 `<!DOCTYPE html>`、末尾 `</html>`、未閉じタグなし。

**ファイル名の決定（旧 `_slug()` 互換）:**
書名を NFKC 正規化 → 空白を `_` に → `[^\w\-]`（Python の unicode \w 基準。日本語文字は残る）を `_` に → 連続 `_` を1つに圧縮 → 先頭末尾の `_` を除去 → 80文字まで。
例: 「DIE WITH ZERO 人生が豊かになりすぎる究極のルール」→ `DIE_WITH_ZERO_人生が豊かになりすぎる究極のルール_infographic.html`

**保存先（2箇所）:**
- `infographics/<slug>_infographic.html`（マスター）
- `docs/<slug>_infographic.html`（GitHub Pages 公開用、同一内容のコピー）

**公開URL:**
`https://oshomadesse.github.io/books-summary/<ファイル名をURLエンコード>?openExternalBrowser=1`

## Step 5: データ更新

1. `data/books_read.json` の末尾に追記: `{"date": "YYYY-MM-DD", "title": "...", "author": "...", "category": "..."}`
2. `data/latest.json` を**上書き**で作成（LINE 通知 Action が読む）:

```json
{
  "date": "YYYY-MM-DD",
  "title": "書名",
  "author": "著者",
  "core_message": "核心的メッセージ（60文字以内。超える場合は60文字で切って…を付ける）",
  "infographic_url": "https://oshomadesse.github.io/books-summary/..._infographic.html?openExternalBrowser=1"
}
```

## Step 6: コミット & push

```bash
git add 100_Inbox/Books-YYYY-MM-DD.md infographics/ docs/ data/books_read.json data/latest.json
git commit -m "📚 Daily reading update (<書名>)"
git push origin main
```

- コミットメッセージに `[skip ci]` を**入れない**こと（LINE 通知 Action の発火に必要）。
- push が reject された場合は `git pull --rebase origin main` して再 push（最大3回）。
- push 完了をもってこの Routine の仕事は終了。Pages 反映待ちと LINE 送信は Action 側が行う。
